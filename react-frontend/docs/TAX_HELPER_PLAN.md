## Reality check on "upload JSON to income tax website"

The Indian Income Tax e-filing portal (incometax.gov.in) **does** accept JSON
uploads — but only for specific use cases:

| Path | Accepts | Notes |
|---|---|---|
| **Online filing** | filled forms in browser | Pre-fills from Form 16 / 26AS / AIS automatically |
| **Offline Excel utility** | JSON generated by their official Excel tool | One JSON per ITR form (1 / 2 / 3 / 4). Schema changes every assessment year. Heavy. |
| **Java utility** | JSON generated by their official Java tool | Same schema as Excel utility |
| **Pre-filled JSON download** | JSON the portal itself produced, edited, re-uploaded | This is a *return trip* — you can't construct it from scratch |

**There is no public REST API** for individual tax filing — no third-party app
can submit a return without going through the portal's authenticated browser
session. So "upload JSON, click authenticate, done" can't literally happen.

What we **can** realistically deliver in three tiers:

---

## Tier 1 — "ITR Helper" report (REALISTIC, ship in v1.4.0)

A new view: **Reports → ITR Helper (India)**.

### Inputs (all already in the app)
- Invoices → business turnover, GST collected
- Expenses → business expenses, GST paid (input credit)
- Receipts → cash received
- Purchases → cost of goods, ITC

### + One new input: bank statement
- Upload **CSV** from any Indian bank (HDFC, ICICI, SBI, Axis, Kotak, IDFC,
  Yes Bank — all export CSV from net-banking).
- Optional: parse **Excel** (xls/xlsx) using SheetJS.
- PDF parsing is significantly harder (each bank's layout differs, no common
  schema, requires OCR for scanned statements). **Defer to Tier 2.**

### What the ITR Helper computes
1. **Gross business turnover** = sum of all tax invoices + bills of supply for
   the financial year.
2. **Gross business expenses** = sum of all expenses + purchases.
3. **Bank-side reconciliation** — match invoices to incoming credits, expenses
   to outgoing debits. Flag unmatched transactions for the user to categorise.
4. **Bank interest income** = sum of credits where description matches
   `INTEREST | INT.PD | INT CR | SAVINGS INTEREST` patterns. (User can
   override per-transaction.)
5. **TDS deducted** — match outgoing debits with description containing
   `TDS | TAX DEDUCTED` and aggregate.
6. **GST summary** — CGST/SGST/IGST collected vs paid (we already have this
   from the GST Returns view).
7. **Net profit before tax** = turnover − expenses.

### Output
- An on-screen summary the user can read like a dashboard.
- **PDF report** — "Income Tax Filing Summary FY 2025-26" with all the
  numbers a CA / the user would type into the ITR form.
- **CSV export** — flat row format for record-keeping or import into Excel.

### What this does NOT do
- Doesn't generate the actual IT Department's ITR JSON schema. That schema is
  ~600 fields, changes yearly, and requires fields we don't track (PAN of
  every payer, Form 16 details, capital gains lots, etc.).
- Doesn't authenticate with the portal. Cannot.
- Doesn't auto-file. Cannot.

### Realistic value
The user opens the ITR-3 / ITR-4 form on the portal and copies numbers from
our PDF report into the fields. Or sends the PDF to their CA. **This is
80% of the value with 5% of the effort** of full automation.

---

## Tier 2 — Bank statement PDF parsing (LATER, v1.5+)

Adds PDF parsing on top of Tier 1. Needs:
- A PDF text extraction library (`pdf.js` in browser, `pdf-parse` server-side).
- Per-bank parsers for the top 8 Indian banks. Each is ~50–100 lines of regex.
- A fallback "manual mapping" UI when the format isn't recognised.
- OCR (Tesseract) for scanned statements. Heavy — only worth it if many users
  have image-only statements.

**Estimate: 2–3 weeks of focused work.**

---

## Tier 3 — ITR-4 (Sugam) JSON generation (FUTURE, opt-in)

ITR-4 is the simplest ITR form: presumptive income under 44AD/44ADA/44AE. Most
of our small-business / freelancer users qualify.

The Excel utility for ITR-4 produces JSON with ~80 fields (vs ~600 for ITR-3).
We could target it specifically:

- User clicks "Generate ITR-4 JSON".
- We pre-fill: PAN, business name, address, gross receipts (= our turnover),
  presumptive profit @ 6% / 8% / 50% per the chosen section.
- User edits any remaining fields.
- We produce a JSON matching the AY 2025-26 Excel utility schema.
- User uploads it to the portal manually (still the "click authenticate, done"
  step the user wanted, just at the upload step rather than via API).

**Caveats:**
- Schema changes every assessment year. We'd need to update annually.
- If the user has *any* income outside ITR-4 scope (capital gains, foreign
  income, second house) the JSON won't validate and they'll have to use ITR-3.
- Worth doing **only if** real users ask after Tier 1 ships.

**Estimate: 1–2 weeks for the AY 2025-26 schema. Recurring annual maintenance.**

## Recommended path

1. **Now (v1.3.0)** — ship everything else from this release; document this
   plan; do not build the tax helper yet.
2. **v1.4.0 (next release)** — build Tier 1: bank CSV import + ITR Helper PDF
   report. Get real users using it.
3. **v1.5.x onward** — add Tier 2 (PDF parsing) and/or Tier 3 (ITR-4 JSON)
   based on actual user feedback. Don't build Tier 3 speculatively — the
   maintenance burden is real.
