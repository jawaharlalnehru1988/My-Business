# Multi-account Payment Support — Design Spec

**Status:** Approved, in implementation
**Target release:** v1.5.0
**Author:** DiceCodes
**Date:** 2026-04-30

## Problem

Each business profile in v1.4.3 stores a single bank account as four flat
fields on the profile: `bankName`, `accountNumber`, `ifsc`, `swift`, plus a
single `upiId`. Users with multiple business bank accounts — common for
Indian SMEs running an INR current account plus a USD EEFC plus a personal
UPI — have to either swap profile or hand-edit Settings between invoices.

## Goals

1. Each business profile can hold N payment accounts.
2. Each account bundles its own bank fields *and* its own UPI ID — the
   bank-details block and the UPI QR on the PDF flow from the same account.
3. The user picks an account per invoice from a dropdown in the Customize
   panel; the choice persists on the invoice record so re-rendering produces
   the same PDF.
4. **No regression** for existing v1.4.3 invoices, profiles, or backups. Old
   data must render byte-identical on the same PDF style.
5. Onboarding (Welcome wizard) stays one-account-simple. Power users grow
   into the manager.

## Non-goals (deferred to v1.5+)

- Per-client default account (Vyapar-style "always use HDFC for Acme Corp").
- Multiple accounts shown on a single PDF ("pay any of these").
- Account-level archive separate from "Inactive" — for now Inactive accounts
  remain editable, just hidden from new-invoice dropdowns.

## Data Model

Each profile gains:

```js
paymentAccounts: Account[]   // ordered; UI order = render order
```

```js
Account = {
  id: 'acc_<timestamp>_<rand>',  // stable, generated by createEmptyAccount()
  label: 'HDFC Current — 1234',  // free-text, shown in dropdown + optional on PDF
  bankName: 'HDFC Bank',
  accountNumber: '50100123456789',
  ifsc: 'HDFC0001234',           // re-labelled per profile country (IBAN / Sort Code / Routing)
  swift: 'HDFCINBB',             // optional, foreign wires
  upiId: 'acme@hdfcbank',        // optional; absence means no QR for this account
  isDefault: false,              // exactly one ⭐ per profile (enforced on save)
  isActive: true,                // hidden from new-invoice dropdowns when false
  notes: '',                     // internal-only, never on PDF
}
```

The **legacy flat fields** (`profile.bankName` / `accountNumber` / `ifsc` /
`swift` / `upiId`) are kept on the profile object indefinitely as
**read-only fallbacks**:

- `getPaymentAccounts(profile)` returns `profile.paymentAccounts` if it
  exists and is non-empty.
- Otherwise it **synthesises** a single legacy account from the flat fields
  with `id: 'legacy'`, `isDefault: true`, `isActive: true`. This synthesis
  is in-memory only — never written back unless the user explicitly edits.
- This lets every existing invoice and profile keep working with zero data
  migration. Invoices with no `selectedAccountId` resolve to the synthesised
  legacy account; their PDFs render byte-identical to v1.4.3.

## Invoice-form Integration

`invoiceOptions` gains:

```js
selectedAccountId: 'acc_...' | null  // null ⇒ use active profile's default
showAccountLabel: false              // toggle to print "Pay via: <label>" on PDF
```

The dropdown lives in the **Customize → Footer** group (next to *Bank
Details*, *UPI QR*, *Signature*). It lists only `isActive` accounts of the
**active business profile**. Reads:

1. `localStorage.gst_lastUsedAccountId_<profile.id>` if it still resolves to
   an active account on the active profile,
2. else the active profile's `isDefault: true` account,
3. else first active account,
4. else null (no accounts → no bank block, no QR — same as today's empty
   bank-name case).

On invoice save, `gst_lastUsedAccountId_<profile.id>` is updated to whatever
the user picked.

## Settings UI

New top-level card **"Payment Accounts"** in Settings, positioned between
Region Preference and Modules. Independent of the Business Profile form so
adding/editing accounts doesn't dirty the profile.

Per account row:
- ⭐ badge if default, plain otherwise
- Label (bold)
- Masked account number: `••••6789`
- Bank name + country-aware bank label (`IFSC HDFC0001234`)
- UPI ID inline if set (`📱 acme@hdfcbank`)
- "Inactive" greyed-out style if `isActive: false`
- Action buttons: ⭐ Set default · ↑ ↓ Move · Edit · Activate/Deactivate · Delete

Empty state: "No accounts yet. Add the first one — it's auto-marked ⭐
Primary."

Migration banner (shown when `profile.paymentAccounts` is empty/missing
*and* any legacy flat field is set): "Imported your existing bank details
as the first account. Edit it or add more below." Clicking *Import &
continue* writes the synthesised account into `paymentAccounts`. The
banner stops showing once `paymentAccounts.length > 0` (covers both
"clicked the button" and "user manually added an account").

Add/Edit form: label, bank name, account number, IFSC (country-aware
label), SWIFT, UPI ID, notes (textarea). Soft validation on UPI ID via
regex `^[\w.\-]+@[\w.\-]+$` — warning only, never blocks save.

## PDF Rendering

`InvoicePreview` resolves the account once at render:

```js
const account = getAccountById(profile, options.selectedAccountId);
```

The existing bank-details block reads from `account` (was `profile`). The
UPI QR `useEffect` reads `account?.upiId` (was `profile.upiId`); if absent,
no QR (same fallback path as today's empty `upiId` case).

When `options.showAccountLabel` is true and `account.label` is set, a small
"Pay via: <label>" line renders above the bank-details block.

Country-aware bank label (`bankCC.bankLabel` → IFSC / IBAN / Sort Code /
Routing) still flows from `profile.country` — unchanged.

## Welcome Wizard

The bank step keeps its current four flat inputs. On *Finish*, the wizard
ALSO writes:

```js
profile.paymentAccounts = [{
  id: createEmptyAccount().id,
  label: profile.bankName || 'Default account',
  bankName, accountNumber, ifsc, swift, upiId,
  isDefault: true, isActive: true, notes: '',
}];
```

Net result: a brand-new user has one account auto-marked Primary, ready to
use, no extra wizard steps.

## Backup / Restore

`paymentAccounts` is an array property of the profile object, which already
rides along under the existing "Active business profile" / "All business
profiles" checkboxes in the granular Export/Import modal. No changes
needed to the backup pipeline.

## Backward Compatibility

| Scenario | Behaviour |
|---|---|
| Old profile (no `paymentAccounts`), open invoice form | Dropdown shows synthesised "Default account" from legacy fields. Pick it or leave default → PDF identical to v1.4.3. |
| Old invoice (no `selectedAccountId`), reopen for editing | Falls through to active profile's default → PDF identical. |
| Old invoice, profile later gained `paymentAccounts` | Falls through to current default account. PDF shows whatever the current default's details are — same behaviour as v1.4.2 "profile rename auto-propagates". |
| Old backup imported | Profile arrives with flat fields only → next time it's loaded, synthesised account is offered via migration banner. User clicks *Import & continue* → array written → flat fields retained as backup. |
| Backup from v1.5+ imported into v1.4.x | `paymentAccounts` array silently ignored by v1.4.x; flat fields (which v1.5 keeps in sync for the default account) drive everything. Graceful one-way downgrade. |

## Files Touched

- `src/utils.js` — extend existing helpers: `maskAccountNumber()`,
  `reorderAccounts(list, fromIdx, toIdx)`, `getActiveAccounts(profile)`,
  `setDefaultAccount(profile, accountId)`. The `getPaymentAccounts` /
  `getDefaultAccount` / `getAccountById` / `createEmptyAccount` helpers
  added during the brainstorm phase stay.
- `src/components/SettingsView.jsx` — new Payment Accounts card; remove the
  inline bank fields from the Business Profile card. The flat-field
  inputs stay in the profile state object for the migration banner case.
- `src/components/InvoiceGenerator.jsx` — Customize → Footer dropdown;
  `selectedAccountId` + `showAccountLabel` defaults in `DEFAULT_OPTIONS`;
  last-used memory write on save.
- `src/components/InvoicePreview.jsx` — resolve account once; bank block +
  UPI QR read from `account`; optional `Pay via: <label>` line.
- `src/components/WelcomeGuide.jsx` — on finish, mirror flat fields into
  `paymentAccounts` array.
- `src/userGuideContent.js` — add a "Payment Accounts" subsection.
- `CHANGELOG.md` — v1.5.0 entry.
- `package.json` — version → 1.5.0.

## Testing Plan

Manual smoke-test matrix (no Vitest in this repo yet):

1. Fresh install, run Welcome wizard with one account → invoice → PDF shows account fields and UPI QR.
2. v1.4.3 backup imported → migration banner shown → click *Import & continue* → account appears in manager.
3. v1.4.3 invoice opened from Bills list → PDF byte-comparable to v1.4.3 build (manual diff of bank block + QR).
4. Add second account, mark Primary → new invoice defaults to it. Pick first manually → preserved in `selectedAccountId`.
5. Mark default account Inactive → falls back to next active account on new invoices.
6. Delete the selected account on an existing invoice → falls through to current default; banner: "The account this invoice used was removed; falling back to <default>."
7. Two business profiles, each with its own accounts → switching profile changes dropdown contents without leakage.
8. Foreign profile (UAE) → bank label reads "IBAN" not "IFSC". SWIFT field visible. UPI hidden (non-INR).
9. PDF in all 3 styles (classic / modern / minimal) — render once per style to confirm no layout regression.
